# 4. Нефункциональные требования (Non-Functional Requirements)

## 4.1 Общее описание

В данном разделе описаны нефункциональные требования к backend-системе **RoadVision AI**.  
Они определяют характеристики системы, не связанные напрямую с бизнес-логикой, такие как производительность, безопасность, масштабируемость и надёжность.

**Формат идентификатора:** `NFR-[КАТЕГОРИЯ]-[НОМЕР]`

---

## 4.2 Производительность (Performance)

| ID | Требование                                                                                |
|----|-------------------------------------------------------------------------------------------|
| **NFR-PERF-01** | API должен обрабатывать не менее 50 запросов в секунду при стандартной нагрузке           |
| **NFR-PERF-02** | Время ответа API (без учёта AI обработки) не должно превышать 300 мс                      |
| **NFR-PERF-03** | Обработка изображения должна занимать не более 3 секунд                                   |
| **NFR-PERF-04** | Обработка видео должна выполняться асинхронно без блокировки API                          |
| **NFR-PERF-05** | Задержка в real-time режиме не должна превышать 2–3 секунд                                |
| **NFR-PERF-06** | Система должна поддерживать одновременную обработку нескольких задач через очередь (Celery) |

---

## 4.3 Масштабируемость (Scalability)

| ID | Требование |
|----|-----------|
| **NFR-SCALE-01** | Архитектура должна поддерживать горизонтальное масштабирование backend-сервисов |
| **NFR-SCALE-02** | AI обработка должна быть вынесена в отдельные воркеры |
| **NFR-SCALE-03** | Система должна поддерживать переход с SQLite на PostgreSQL |
| **NFR-SCALE-04** | Система должна позволять добавление новых моделей AI без изменения API |
| **NFR-SCALE-05** | Количество RTSP потоков должно масштабироваться за счёт увеличения worker-процессов |

---

## 4.4 Надёжность и отказоустойчивость (Reliability)

| ID | Требование |
|----|-----------|
| **NFR-REL-01** | Система должна корректно обрабатывать ошибки AI-модели |
| **NFR-REL-02** | При падении worker-процесса задача должна автоматически перезапускаться |
| **NFR-REL-03** | RTSP поток должен автоматически переподключаться при обрыве |
| **NFR-REL-04** | Система должна обеспечивать целостность данных при сбоях |
| **NFR-REL-05** | Все критические ошибки должны логироваться |

---

## 4.5 Безопасность (Security)

| ID | Требование |
|----|-----------|
| **NFR-SEC-01** | Все API должны быть защищены с использованием JWT аутентификации |
| **NFR-SEC-02** | Пароли пользователей должны храниться в зашифрованном виде (hash) |
| **NFR-SEC-03** | Система должна проверять права доступа (RBAC) |
| **NFR-SEC-04** | Только Admin может управлять пользователями |
| **NFR-SEC-05** | Только Operator и Admin могут управлять RTSP камерами |
| **NFR-SEC-06** | Система должна защищаться от загрузки вредоносных файлов |
| **NFR-SEC-07** | Должна быть реализована защита от SQL-инъекций (через ORM) |
| **NFR-SEC-08** | API должен поддерживать ограничение количества запросов (rate limiting) |

---

## 4.6 Удобство использования (Usability)

| ID | Требование |
|----|-----------|
| **NFR-USE-01** | API должен быть документирован с использованием Swagger / OpenAPI |
| **NFR-USE-02** | Все ответы API должны быть в формате JSON |
| **NFR-USE-03** | Ошибки должны возвращаться в понятном формате с описанием |
| **NFR-USE-04** | API должен поддерживать пагинацию |
| **NFR-USE-05** | API должен поддерживать фильтрацию и сортировку |

---

## 4.7 Поддерживаемость (Maintainability)

| ID | Требование |
|----|-----------|
| **NFR-MAINT-01** | Код должен быть структурирован по принципам модульности (apps) |
| **NFR-MAINT-02** | Все компоненты системы должны быть слабо связаны |
| **NFR-MAINT-03** | Система должна поддерживать добавление новых типов дефектов |
| **NFR-MAINT-04** | Система должна поддерживать добавление новых источников данных |
| **NFR-MAINT-05** | Код должен быть покрыт комментариями и документацией |

---

## 4.8 Совместимость (Compatibility)

| ID | Требование |
|----|-----------|
| **NFR-COMP-01** | API должен работать с любыми клиентами (Web, Mobile) |
| **NFR-COMP-02** | Система должна поддерживать стандартные форматы медиафайлов |
| **NFR-COMP-03** | Система должна поддерживать работу с RTSP камерами |
| **NFR-COMP-04** | Backend должен быть совместим с Linux-средой |

---

## 4.9 Логирование и мониторинг (Logging & Monitoring)

| ID | Требование |
|----|-----------|
| **NFR-LOG-01** | Система должна логировать все запросы к API |
| **NFR-LOG-02** | Система должна логировать действия администратора |
| **NFR-LOG-03** | Система должна логировать ошибки AI |
| **NFR-LOG-04** | Система должна логировать состояние RTSP потоков |
| **NFR-LOG-05** | Логи должны храниться не менее 7 дней |

---

## 4.10 Ограничения (Constraints)

| ID | Ограничение |
|----|------------|
| **NFR-CONSTR-01** | В рамках дипломного проекта используется SQLite |
| **NFR-CONSTR-02** | AI модель ограничена возможностями оборудования |
| **NFR-CONSTR-03** | Максимальное количество одновременно обрабатываемых потоков зависит от CPU/GPU |
| **NFR-CONSTR-04** | Система разворачивается в учебной среде |
