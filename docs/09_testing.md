# 9. Тестирование системы

## 9.1 Общее описание

Целью тестирования является проверка корректности, стабильности и надёжности backend-системы **RoadVision AI**.

В процессе тестирования проверялись:
- корректность работы API
- обработка пользовательских данных
- работа AI-модуля
- стабильность стриминга
- разграничение прав доступа

---

## 9.2 Стратегия тестирования

Тестирование проводилось по следующим направлениям:

- проверка отдельных компонентов (unit)
- проверка взаимодействия компонентов (integration)
- тестирование API (functional testing)
- проверка поведения системы под нагрузкой

---

## 9.3 Виды тестирования

### 9.3.1 Модульное тестирование (Unit Testing)

Проверялись:
- модели (users, defects, cameras)
- сериализаторы
- бизнес-логика

**Пример:**
- создание пользователя
- валидация данных
- корректность сохранения дефекта

---

### 9.3.2 Интеграционное тестирование

Проверялось взаимодействие:

- API ↔ база данных
- API ↔ AI-модуль
- API ↔ streaming сервис

**Пример:**
- загрузка изображения → создание записи → запуск AI
- запуск камеры → создание stream session

---

### 9.3.3 Функциональное тестирование API

Проверялись:

- корректность HTTP-методов (GET, POST, PATCH, DELETE)
- соответствие ответов ожидаемым
- обработка ошибок

---

### 9.3.4 Тестирование ролей (RBAC)

Проверялись ограничения доступа:

| Роль | Проверка |
|-----|--------|
| User | не может управлять камерами |
| Operator | может запускать/останавливать стрим |
| Admin | имеет полный доступ |

---

### 9.3.5 Нагрузочное тестирование

Проверялась устойчивость системы при одновременных запросах.

Параметры:
- 20–50 запросов одновременно

---

## 9.4 Инструменты тестирования

Используемые инструменты:

- **Django Test Framework** — написание unit и integration тестов  
- **Postman** — ручное тестирование API  
- **Swagger / Redoc** — визуальное тестирование  
- **cURL** — быстрые проверки эндпоинтов  

---

## 9.5 Тестовые сценарии API

### Сценарий 1 — Авторизация

**Запрос:**
POST /api/auth/login/

**Ожидаемый результат:**
- статус 200
- возврат JWT токена

---

### Сценарий 2 — Загрузка изображения

**Запрос:**
POST /api/images/

**Ожидаемый результат:**
- статус 201
- изображение сохранено
- запущена AI-обработка

---

### Сценарий 3 — Получение дефектов

**Запрос:**
GET /api/defects/

**Ожидаемый результат:**
- статус 200
- возвращается список дефектов

---

### Сценарий 4 — Доступ без авторизации

**Запрос:**
GET /api/defects/

**Ожидаемый результат:**
- статус 401 Unauthorized

---

### Сценарий 5 — Проверка прав доступа

**Сценарий:**
User пытается удалить камеру

**Ожидаемый результат:**
- статус 403 Forbidden

---

## 9.6 Тестирование AI-модуля

Проверялись:

- обнаружение дефектов на изображениях
- корректность bounding box
- влияние качества изображения

**Результат:**
AI-модель успешно обнаруживает основные типы дефектов с приемлемой точностью.

---

## 9.7 Тестирование стриминга

Проверялись:

- подключение к RTSP-потоку
- запуск и остановка стрима
- восстановление соединения при обрыве

**Результат:**
Система корректно обрабатывает поток и восстанавливает соединение.

---

## 9.8 Нагрузочное тестирование

Результаты:

- API обрабатывает до 50 запросов одновременно
- среднее время ответа < 300 мс
- критических ошибок не выявлено

---

## 9.9 Обработка ошибок

Проверялись сценарии:

- загрузка некорректного файла
- неверные параметры запроса
- отсутствие обязательных полей

**Результат:**
Система возвращает корректные HTTP-коды ошибок (400, 404, 500) и информативные сообщения.

---

## 9.10 Логирование

Проверялась работа логирования:

- запросы API
- ошибки AI
- события стриминга

---

## 9.11 Вывод

В результате проведённого тестирования установлено, что система:

- корректно реализует заявленный функционал
- устойчива к ошибкам
- обеспечивает разграничение прав доступа
- демонстрирует стабильную работу при нагрузке

Система готова к демонстрации и дальнейшему развитию.
